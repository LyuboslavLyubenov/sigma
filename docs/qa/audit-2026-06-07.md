## Executive summary

The remediation branch did its job: **every P0/high issue from 2026-06-03 that was in remediation scope is confirmed fixed** — the unauthenticated ETL write path (the prior lone *critical*) is closed (ETL is cron-only, `workers_dev=false`, no HTTP entry), admin auth fails closed, the three CSV exports are BOM-prefixed and formula-guarded, keyset paging is direction-aware with a sort allowlist (no more `?sort=` 500s or fabricated ranks), and the SQL layer remains fully parameter-bound (a dedicated injection sweep found **zero** SQLi). The CSP/nonce posture is sound in production and **the "no CSP on `/`" signal does not reproduce in code** — it is an artifact of the Vite dev server bypassing the `apps/web/workers/app.ts` entry that hardens every response.

The residual risk has shifted to four themes:

1. **Person-data controls (P0) — the operator's *own* C1 safeguards aren't implemented.** The operator's plan (C1) deliberately publishes ЕТ/individual names as *lawful* (ТЗ чл. 59 + public РОП/ТР + Art 6(1)(e)/(f)) — so this is **not** a lawful-basis dispute. But C1's binding caveat requires single-natural-person pages to stay **`noindex`** and *"not more exposed than РОП"*, plus a real Art 21 path. All three are missing: the `noindex` control **is dead code** (it gates on `bidders.legal_form`, empty for all 16 549 bidders, so it never fires), ~537 ЕТ person-name profiles are indexable, searchable **and advertised in the sitemap** (4 consortium "prose" profiles also render full names + ownership shares verbatim and are equally un-`noindex`'d), and there is **no Art 21 objection path, privacy notice, or controller identity** anywhere on the site. So the P0 work is *implementing C1's caveat*, not redacting.

2. **EOP-migration data-integrity regressions.** The cut to the EOP open-data feed (`eop:%` source prefix) left two scripts still scoped to the retired `admin:%` prefix: the **FX pipeline is broken** (`fx_rates` empty → 90 foreign-currency contracts silently dropped from every € total, including the `93,1 млрд €` headline), and `load-eop.mjs --apply` with a multi-day window **wipes the entire historical corpus**, not just the window — and exits `0` on partial fetch failure after the wipe.

3. **Missing public-facing compliance scaffolding.** No accessibility statement / feedback mechanism / complaint link (ЗЕУ чл. 58г — **still applies**: МИДТ operates the site, and D1 waives NIS2/state-cloud, *not* the Web Accessibility Directive), no impressum (ЗЕТ чл. 4), no user-visible open-data licence/attribution. _(The NIS2 items — incident runbook, D1 backup/DR, `observability=false` — are **downgraded to best-practice** per the operator's D1 determination that NIS2-essential-entity obligations do not bite; still cheap and worth doing.)_

4. **A few fail-open / param-DoS edges.** The contracts `bidder` filter fails *open* on a bad slug (returns all 351 k rows under a "filtered" banner); unbounded `sector`/`year` multi-params 500 the list pages in one request. _(The unwired analytics layer / empty `risk_scores` is **expected — deliberately parked per D2** and honestly disclosed on `/methodology`; only its feeder bugs (`cartel.ts`) remain Tier-A fixes per D2.3.)_

The underlying engineering is otherwise in good shape: clean parameterized SQL, sound keyset pagination, React auto-escaping throughout, strong native accessibility on the hand-written pages (skip link, `lang=bg`, focus management, `<main>`/landmarks, `scope=col` tables, Sankey `role=img` + data-table alternative — **all live-verified**, zero console errors, no 320 px overflow, no invisible-focus steps).

## High-severity findings (8 confirmed)

### Regulatory / person-data (P0)

**H-01 · ЕТ sole-trader natural-person names are indexed & searchable — the `noindex` minimisation guard never fires.** `apps/web/app/routes/company.tsx:13-35` gates `noindex` on `isSingleNaturalPersonProfile(kind, legalForm)`, which returns `false` whenever `!legalForm` (line 14). `getCompany` reads `legalForm` from `bidders.legal_form` (`packages/db/src/queries/details.ts:183`), which is empty for **all 16 549** bidders. So ~537 ЕТ profiles whose `bidders.name` embeds a full personal name (`ЕТ ДРИФТ - НИКОЛАЙ КИРОВ`, …) are indexable, in `search_index`, and emitted in the company sitemap. **Breaches the operator's own C1 caveat** ("keep single-natural-person pages `noindex`; not more exposed than РОП") — *not* a lawful-basis dispute (C1 publishes these names lawfully). _Fix:_ detect natural-person profiles structurally (the `ЕТ ` prefix / `… - <Name>` pattern), drive both `noindex` **and** sitemap exclusion from one shared predicate, and add a DQ assertion that fails if `legal_form` is 100 % NULL so a dead guard can't ship again.

**H-02 · Consortium "prose" profiles render verbatim free-text with full names + ownership shares.** `format.ts:151-164` (`PROSE_RE`) routes matching consortium names to a verbatim `raw` blockquote rendered at `company.tsx:118-159` (`whiteSpace:'pre-wrap'`, hint: "Запазваме описанието както е в обявата"). This is tender-notice data (already public via АОП — *not* the C2-removed чл. 63 ЗМИП UBO data), so likely C1-lawful to hold — **but** these consortium pages are **not `noindex`'d** (`isSingleNaturalPersonProfile` returns `false` for `kind==='consortium'`) and are sitemapped, so the C1 caveat's proportionality intent is unmet. _Fix:_ extend the H-01 `noindex`/sitemap-exclusion predicate to consortium pages bearing person names; consider masking the share-percentage prose.

**H-03 · No privacy notice, controller/DPO identity, or Art. 21 objection mechanism.** `routes.ts` has no `/privacy`, `/gdpr`, `/contact`, `/impressum`; `SiteFooter.tsx:6-14` is a source line + Методология link only. GDPR Arts 13–14 (incl. Art. 14 source disclosure) & 21 unmet for the named individuals above. _Fix:_ add a `/privacy` route (controller + contact, registry sources, lawful basis, retention, Art. 21/erasure) linked site-wide; this is a code-deliverable that unblocks the P0 checklist.

**H-04 · No accessibility statement / feedback mechanism / complaint link (ЗЕУ чл. 58г).** No `/accessibility` route; methodology §10 is a *data-correction* note, not a декларация за достъпност. A hard, audit-visible gap for a PUB body; МЕУ can issue binding remediation. _Fix:_ author the 2018/1523-model statement (honest conformance level against EN 301 549 / WCAG 2.1 AA), route it, link from the footer, add a feedback contact + МЕУ enforcement link.

### Engineering (correctness / security / data-integrity)

**H-05 · `load-eop.mjs --apply` with a multi-day window wipes the ENTIRE `eop:<cat>` corpus.** `scripts/load-eop.mjs:509-512` emits `DELETE … WHERE source LIKE 'eop:<cat>:%'` for the multi-day branch — unscoped to `--from/--to`. `node load-eop.mjs --from=2024-01-01 --to=2024-01-31 --apply` deletes 2020–2025 and reloads only Jan 2024. Catastrophic on the new canonical source; split DELETE/INSERT across `--file` executions means an interrupted run leaves the corpus deleted and partly reloaded. _Fix:_ scope the wipe to the requested days, or require an explicit `--full` flag for a whole-corpus wipe.

**H-06 · `load-eop.mjs` exits `0` on partial/total fetch failure — after `--apply` has already wiped.** `:636-640` only fails when *zero* day-cats complete; a 403/`_missing` day is counted as *completed* (`:407-442,577`). Per the documented foreign-egress-403 reality, a windowed reload can fetch nothing, wipe, and report success. _Fix:_ `process.exitCode = 1` on any failed day; better, abort *before* emitting the DELETE if any requested day failed.

**H-07 · `companies.csv` silently ignores sector/year/EU filters — exports the full 16.5 k corpus.** `companies.csv.tsx:7-16` forwards only `sort/kind/count`; the page link (`companies.tsx:77`) preserves sector/year/eu, so a filtered URL downloads everything. Partial regression of the `493efc2` filter-scoping fix (authorities/contracts CSV do call `source(p)`; companies doesn't). _Fix:_ forward `sectors/years/eu` to `streamCompaniesCsv`; add a row-count regression test. _(Same root cause as `code-web-routes#1`.)_

**H-08 · Sector/Year/EU filters run a full 351 k-row scan, executed twice per request.** `packages/db/src/queries/companies.ts:52-79` flips to an on-the-fly `FROM contracts JOIN bidders JOIN tenders … GROUP BY` when those (user-clickable) filters are set, and `listCompanies` runs the page query **and** an identical `countCompanies` in parallel (`:111-117`). Multi-second cold; under concurrency this saturates the single D1/worker — **this is what crashed `workerd` during the audit.** On production D1 it risks per-query row-scan/time limits. _Fix:_ stop double-running the heavy aggregate for COUNT; precompute `company_sector/year_totals` rollups refreshed in ETL.

> **Also park-fix with H-08:** **DoS · unbounded `sector`/`year` multi-params** (`apps/web/app/lib/filters.ts:12-19` `getMulti` has no length cap) → `?sector=1,2,…(≥100)` triggers D1 "too many SQL variables" → unhandled 500 on `/contracts`, `/companies`, `/authorities` from one trivial GET (and header-only CSV truncation). _Fix:_ cap `getMulti` to ~50 and intersect `sectors` against the 45 known CPV divisions / `years` against the known range, like `procedure`/`kind` already are. **(Confirmed high; grouped here as it shares the filter surface.)**

## Methodology note

The browser phase was partially disrupted mid-run: under ~14 concurrent headless-Chromium sessions plus the H-08 full-table scans, the Cloudflare dev runtime (`workerd-linux-arm64`) took a **SIGBUS** native crash, after which every SSR route 500'd. The adversarial verifiers correctly caught this (one "site down" finding downgraded to info, one refuted as environmental). The three browser areas blocked by the crash (home/nav/search, flows/methodology, cross-page a11y/responsive) were **re-run after restarting a single clean dev server** — those results populate the browser table above. The lesson for the project's own CI/load posture: the local dev runtime is fragile under concurrency, and H-08's double full-table scan is the realistic production analogue.
